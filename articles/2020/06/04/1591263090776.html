核心内容

（可扫码体验）

🎉 子琦老师敲黑板

PWA 并不是单指某一项技术，你更可以把它理解成是一种思想和概念，目的就是对标原生 app，将 Web 网站通过一系列的 Web 技术去优化它，提升其安全性，性能，流畅性，用户体验等各方面指标，最后达到用户就像在用 app 一样的感觉。

为什么要用 service woker

  • 你只需要关心 W3C 的 Web 标准，不用关心各种 Native APP 的代码。
  • 用户可以在安装应用之前先试用。
  • 不需要使用各种应用商店来分发应用，也不用关心应用发布时奇怪的审核标准以及应用内购的平台抽成。
  • 应用程序更新是自动进行的，无需用户交互，使用体验更为平滑。
  • “安装”过程很快，只需要在主屏幕上添加一个图标即可。
  • 轻量级安装——你只需要缓存几百 KB 的数据即可。
  • 所有的数据传输必须使用安全的 HTTPS 连接
  • 渐进式 Web 应用可以离线缓存数据，并且会在重新连接互联网时重新同步数据。

看着 PWA 应用有如此之多的好处，小伙伴们个个都心神专注地听得不亦乐乎 😋 。

前端缓存技术方案

接着，为了更好地让大家理解 service woker 的出现，子琦老师给大家讲解了当前前端领域内的三种缓存技术方案：

  • http 缓存
  • 浏览器缓存
  • 应用缓存

并全面地给大家讲解了各种常见与不常察觉的缓存方式，让小伙伴们醍醐灌顶，接受到了对前端缓存方式认知的全面洗礼。

Service Worker

一个服务器与浏览器之间的中间人角色，如果网站中注册了 SW 那么它可以拦截当前网站所有的请求，进行判断（需要编写相应的判断程序），如果需要向服务器发起请求的就转给服务器，如果可以直接使用缓存的就直接返回缓存不再转给服务器。从而大大提高浏览体验。

service woker 本质上也是浏览器缓存资源用的，只不过他不仅仅是 cache，也是通过 worker 的方式来进一步优化。

它基于 h5 的 Web worker，所以绝对不会阻碍当前 js 线程的执行

生命周期

使用条件

service woker 是基于 HTTPS 的，因为 service worker 中涉及到请求拦截，所以必须使用 HTTPS 协议来保障安全。如果是本地调试的话，localhost 是可以的。

过程

  • Service Worker 在 sw.js 进行注册，首次注册前会进行分析，判断加载的文件是否在域名下，协议是否为 HTTPS 的，通过这两点则成功注册。
  • service Worker 开始进入下一个生命周期状态 install， install 完成后会触发 service Worker 的 install 事件。 如果 install 成功则接下来是 activate 状态， 然后这个 service worker 才能接管页面。
  • 当事件 active 事件执行完成之后，此时 service Worker 有两种状态，一种是 active，一种是 terminated。
  • active 是为了工作，terminated 则为了节省内存。当新的 service Worker 处于 install/waitting 阶段，当前 service Worker 处于 terminated，就会发生交接替换。或者可以通过调用 self.skipWaiting() 方法跳过等待。
  • 被替换掉的原有的 service Worker 到 Redundant 阶段，在 install 或者 activating 中断的也会进入 Redundant 阶段。
